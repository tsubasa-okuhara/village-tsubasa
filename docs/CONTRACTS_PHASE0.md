## 1. 介護業界特有の論点（方針に影響する前提知識）

### 1.1 重要事項説明書・契約書の電子化は可能か

介護保険法関係（指定居宅サービス等の事業の人員、設備及び運営に関する基準）は **2021年の改正以降、電磁的方法による交付が明示的に認められている**。ただし以下の要件を満たす必要がある:

- 利用者（または家族）の **事前承諾** を得る（承諾自体も電磁的に取得可）
- **書面に出力できる形式** で提供する（PDF 等）
- 改ざん防止のため **電子署名＋タイムスタンプ** を施す（推奨）
- 承諾を撤回された場合は書面交付に戻す

介護現場での実運用では、「紙で読んで押印」→「タブレットで読んで電子署名」の移行期であり、**両方を選べる設計**にしておくのが安全。

### 1.2 国定型様式との関係

利用者関連（重要事項説明書・契約書）は、都道府県・市町村・国のガイドラインに定型があり、事業所が内容を自由に書き換えることは実務上少ない。つまり:

- **テンプレート本文は PDF/DOCX で事業所が持ち込む**前提で設計する（本文生成 AI は不要）
- **差し込み項目**（利用者名・住所・契約日・料金表・担当者名）だけを変数化する
- 本文の改訂時はテンプレートを **バージョン管理**（旧バージョンで締結済みの契約は改訂の影響を受けない）

### 1.3 代理人署名の必須要件（利用者契約）

要介護者の場合、本人が判断能力を持たない／身体的に署名困難であることが多い。運用として必要:

| 役割 | 署名権限 | 本人確認 | 備考 |
|---|---|---|---|
| 利用者本人 | あり | 必須 | 判断能力があれば第一選択 |
| 成年後見人 | あり（代理） | 登記事項証明書を添付 | 後見登記が前提 |
| 家族（配偶者・子）| あり（代理） | 続柄確認書類を添付 | 「事実上の代理」運用の事業所多し |
| 同意者（家族）| 同意の署名のみ | 任意 | 本人署名＋家族同意の二段構え |

→ **契約1件に対して複数の署名者を紐付けられるモデル**（`contract_parties` テーブル）が必要。

## 2. 外部署名 API 比較（GMOサイン vs クラウドサイン）

### 2.1 基本情報

| 項目 | GMOサイン | クラウドサイン |
|---|---|---|
| 運営会社 | GMOグローバルサイン・HD | 弁護士ドットコム |
| 国内シェア | No.1（導入企業数350万社） | No.1（利用者数ベース） |
| 認証方式 | 当事者型（実印相当）＋ 立会人型（メール認証）両対応 | 立会人型中心（実印相当はオプション） |
| 電子帳簿保存法 | 対応 | 対応 |
| タイムスタンプ | 時刻認証局（TSA）連携済み | 時刻認証局（TSA）連携済み |
| 10年保管 | 標準対応 | 標準対応 |
| 介護業界実績 | 豊富（介護・医療・福祉の導入事例あり） | あり（介護事業向け比較記事でも常連） |

### 2.2 API 仕様（公開情報ベース）

| 項目 | GMOサイン | クラウドサイン |
|---|---|---|
| 利用条件 | スタンダードプラン以上 ＋ **API仕様書開示・検証環境・技術支援に別途費用** | スタンダードプラン以上／エンタープライズでも可（2022-08〜） |
| プロトコル | REST | REST |
| 認証方式 | 運営事務局発行の **認証トークン** | **Client ID** 発行→`POST https://api.cloudsign.jp/token` で Bearer 取得（有効期限1時間、client_secret 不要という特殊仕様） |
| 仕様書の公開度 | ❌ 非公開（NDA＋費用後に開示） | ✅ **SwaggerHub で公開**（`app.swaggerhub.com/api/CloudSign/cloudsign-web_api`） |
| Webhook | ✅ あり（署名完了時に指定URLへ JSON POST） | ✅ あり |
| ファイル上限 | 要確認 | 1ファイル 20MB |
| サンドボックス環境 | 別途費用 | 一般アカウントで試行可（送信料は発生） |

### 2.3 料金感（公開情報のみ）

| 項目 | GMOサイン | クラウドサイン |
|---|---|---|
| 基本プラン | スタンダード以上。金額は要問い合わせ | プラン別（スタンダード／コーポレート／ビジネス／エンタープライズ） |
| 1送信あたり | **110円（税込）〜**（公表） | 200円前後と言われるが要問い合わせ |
| API 初期費用 | 仕様書・検証環境・技術支援に別途費用 | Web API は各プランで標準提供 |

### 2.4 介護事業所視点での比較結論（暫定）

| 観点 | 推奨 | 理由 |
|---|---|---|
| **開発着手しやすさ** | クラウドサイン | API 仕様書が公開されていて検証前に設計可。トークン取得が単純 |
| **1送信料金の透明性** | GMOサイン | 110円〜と公表。件数試算しやすい |
| **介護業界実績** | やや GMOサイン | 医療・福祉・介護の導入事例ページが充実。ただしクラウドサインも介護向け比較記事で紹介実績あり |
| **当事者型署名（実印相当）** | GMOサイン | 雇用契約・重要事項説明書で実印相当が必要になる場面で強い |
| **利用者側の体験（高齢者）** | ほぼ同等 | どちらもメール＋SMS認証が可能。UIはどちらも自社アプリ内完結にすれば差は出ない |

**Claudeからの推奨**: **クラウドサインで試作 → 年間件数が見えてきたら GMOサインも含めて再評価** という2段構えを提案。理由:

1. クラウドサインは **仕様書公開** なので、NDA・費用をかける前に `CONTRACTS_DESIGN.md` を書き切れる
2. `functions/src/contracts/providers/` にプロバイダ抽象層を作っておけば、後から GMOサインに差し替えまたは併用可能
3. 最初の半年は件数が少ないはず（月数十件規模）→ 料金差は吸収できる範囲

ただしこれは **机上比較**。実際に決める前に以下を確認するタスクを後工程に入れる:

- GMOサイン営業へ「介護事業・月◯件想定、API連携、代理人署名あり」で見積もり依頼
- クラウドサイン営業へ同条件で見積もり依頼 + SwaggerHub の仕様書を読み込み
- 両者のサンドボックス／テスト環境で本人確認フロー（特に代理人）を実機検証

## 3. 提案アーキテクチャ（コード配置のレベル）

```
village-tsubasa/ (既存リポ)
├── functions/src/
│   ├── index.ts                    # 既存: /api ルータに /contracts/* を追加
│   └── contracts/                  # 【新規】契約モジュール
│       ├── router.ts               # /api/contracts/* ルータ
│       ├── handlers/               # 各エンドポイント実装
│       │   ├── list.ts
│       │   ├── create.ts
│       │   ├── sign-request.ts
│       │   ├── webhook.ts          # 外部APIからの完了通知受信
│       │   └── download.ts         # 署名済みPDF取得
│       ├── services/
│       │   ├── template.ts         # テンプレート差し込み
│       │   └── audit.ts            # 監査ログ書き込み
│       ├── providers/              # 【抽象化層】外部署名API
│       │   ├── types.ts            # 共通インタフェース
│       │   ├── cloudsign.ts        # クラウドサイン実装
│       │   └── gmosign.ts          # GMOサイン実装（後から追加）
│       └── schemas/                # リクエスト/レスポンス zod スキーマ
│
├── public/
│   ├── contracts/                  # 【新規】ヘルパー向け「雇用契約」画面
│   │   ├── index.html              # 一覧
│   │   ├── sign.html               # 署名画面（外部API の iframe or redirect）
│   │   └── viewer.html             # 締結済み契約の閲覧
│
├── sql/
│   └── create_contracts.sql        # 【新規】契約系テーブル

user-schedule-app/ (別リポ、GitHub Pages)
└── contracts/                       # 【新規】利用者向け「契約書」画面
    ├── index.html
    └── sign.html                    # 重要事項説明書の同意フロー
                                     # Supabase anon key ではなく、Functions の
                                     # /api/contracts/public/* を叩く（RLS節約）

village-admin/ (別リポ)
└── public/
    └── contracts.html              # 【新規】管理者向け契約一覧・テンプレート管理
```

### 3.1 既存認証の流用方針

| アプリ | 既存認証 | 契約機能での扱い |
|---|---|---|
| village-tsubasa | Firebase 側でメール特定（`helper_email`） | 既存フローをそのまま使う。ヘルパーは自分宛契約だけ見える |
| village-admin | `admin_users` allow-list | 既存 `adminAuth` ミドルウェアを契約系 API にも適用 |
| user-schedule-app | Supabase anon key 直叩き | **契約系だけは Functions 経由に切替**（RLS+匿名鍵では代理人署名や監査要件が厳しい） |

### 3.2 外部API呼び出しの境界

```
┌─────────────┐   1. 契約作成    ┌──────────────────┐
│  村ひろば    │ ──────────────→ │ /api/contracts   │
│ helper画面   │                   │ Functions         │
└─────────────┘                   └───┬──────────────┘
                                       │ 2. 署名依頼
                                       ↓
                                ┌──────────────────┐
                                │ providers/       │
                                │ cloudsign.ts     │
                                └───┬──────────────┘
                                    │ 3. REST
                                    ↓
                            ┌─────────────────┐
                            │ クラウドサイン  │
                            │      API         │
                            └───┬─────────────┘
                                │ 4. Webhook（完了通知）
                                ↓
                         ┌──────────────────┐
                         │ /api/contracts/  │
                         │     webhook      │
                         └──────────────────┘
```

→ 外部API の認証情報・呼び出しは **providers/ 以下に完全に閉じ込める**。Functions のほかの層は「署名者リスト」「テンプレート」などのドメイン概念だけ扱う。

## 4. MVP スコープ（Phase 1〜4 で作るもの）

### 4.1 対象契約の3分類と優先順位

| # | 分類 | 対象者 | 件数感（年間） | 国定型 | 代理人 | MVP優先 |
|---|---|---|---|---|---|---|
| 1 | **ヘルパー雇用契約** | helper_master | 20〜40件＋毎年の更新通知 | なし（自社） | 不要 | ★★★ 最優先 |
| 2 | **業務委託・秘密保持** | 他事業所・協力会社 | 5〜10件 | なし（自社） | 不要 | ★★ |
| 3 | **利用者 重要事項説明書・契約書** | 利用者116名＋新規 | 20〜30件 + 改訂時全件 | **あり** | **あり** | ★ 先に基盤だけ、本格運用はフェーズ後半 |

### 4.2 MVP で実装する／しない

**実装する**
- 契約テンプレートの登録・バージョン管理（PDF 差し込み型）
- 契約作成（テンプレートを選び、宛先と差し込み値を入れて送信）
- 外部署名API 呼び出し（クラウドサイン）
- Webhook 受信による署名済み状態への更新
- 監査ログ（誰がいつ何をした）
- 代理人署名（利用者契約向け、1契約に対して複数署名者）
- ヘルパー側「雇用契約」タブ、利用者側「契約書」タブ、管理者側「契約管理」画面

**実装しない（後回し）**
- GMOサイン対応（providers/ 抽象は用意する）
- 紙契約との並行運用ワークフロー（紙はこれまで通り）
- 契約更新の自動通知ジョブ（手動トリガーで十分）
- AIによる契約書本文のチェック（国定型の重要事項説明書は書式固定のため不要、自社雇用契約は人手レビューで十分）

## 5. RULES.md との整合チェック

| ルール | 該当するか | 対応 |
|---|---|---|
| ルール2：既存テーブル削除・型変更・NOT NULL 化禁止 | 非該当 | 新規テーブルのみ追加（`contracts`, `contract_templates`, `contract_parties`, `contract_signatures`, `contract_audit_log`） |
| ルール3：既存APIの破壊的変更禁止 | 非該当 | `/api/contracts/*` を新規追加のみ |
| ルール4：`SUPABASE_SCHEMA.md` ⚠️/✅ テーブルを触る場合は事前相談 | 一部該当 | `helper_master` と `notifications` を **SELECT のみ** で参照（INSERT なし）。挙動変更なし。念のため設計書で明記 |

| ルール5：作業終了時 CHANGELOG.md に追記 | 該当 | Phase 0 の本メモ作成を CHANGELOG.md に追記予定 |

## 6. Phase 1 以降の具体的な成果物イメージ

### Phase 1: `docs/CONTRACTS_DESIGN.md`
- DB ER 図（mermaid）
- API エンドポイント一覧（メソッド・パス・認可・リクエスト例・レスポンス例）
- 画面遷移図（ヘルパー／利用者／管理者それぞれ）
- 状態遷移（draft → pending_signature → signed → expired, revoked）
- 代理人署名のフロー詳細
- クラウドサイン providers/types.ts のインタフェース定義

### Phase 2: `sql/create_contracts.sql`
新規 5 テーブル（目安）:
- `contracts` — 契約マスタ
- `contract_templates` — テンプレート（PDF本文＋差し込みフィールド定義）
- `contract_parties` — 署名者一覧（本人／代理人／同意者の区別を持つ）
- `contract_signatures` — 各署名者の署名状態・タイムスタンプ・外部API署名ID
- `contract_audit_log` — 監査ログ（append-only）

### Phase 3: `functions/src/contracts/`
- エンドポイント 8 本程度（list / detail / create / sign-request / webhook / download / template-list / template-create）
- providers/cloudsign.ts の実装（認証・ドキュメント作成・署名者追加・送信・Webhook検証）
- Zod スキーマでリクエスト検証

### Phase 4: UI 雛形
- village-tsubasa `public/contracts/` （ヘルパー向け）
- user-schedule-app `contracts/` （利用者向け）
- village-admin `public/contracts.html` （管理者向け）
