# FUTURE_IDEAS — 将来構想とアイデアの蓄積ログ

> **目的**: 日々のチャットや作業中に思いついた「将来こうしたい」「いつかやりたい」を、忘れないうちに吐き出して蓄積する場所。
>
> **運用（RULES.md ルール10 参照）**:
> - 奥原翼さんはチャットで自由に話すだけでOK。粗くてOK、書きっぱなしでOK
> - Claude がアイデアを検出して整理・要約してこのファイルに追記
> - 意図や文脈が曖昧なときは Claude が質問して言語化を助ける
> - 追記は都度告知せず、チャットの最後にまとめて「今日は N 件追加した」と報告
> - 月1回くらい見返して、実行フェーズに入るものは `CURRENT_STATE.md` や別ファイルに昇格、不要なものは **🗄️ 却下アーカイブ** に移動
>
> **セクション構成**:
> - 🧠 **即席メモ**: 軽い思いつき、1〜2行。時系列で追記
> - 📝 **正式エントリ**: 3要素以上（ゴール・手段・期限）を含む構想。構造化して記録
> - 🗄️ **却下アーカイブ**: 実行しないと決めたもの + 判断理由（後から再評価できるように残す）

## 📝 正式エントリ

### 2026-05-06 外部向けマッチングアプリ構想（個人事業主ヘルパー × 利用者の直接マッチ）

**きっかけ**
- 2026-05-06 のチャットで奥原さんから提案
- 既存のビレッジつばさ 3 アプリ群（社内向け）とは別に、地方の個人事業主ヘルパーが利用者と直接マッチして賃金を上げる、Uber / HandyShare 的なプラットフォームを作りたいというビジョン
- 「セルフマッチング Phase 1」（社内向け）とは **明確に別事業** として設計。深掘りは別チャットで進行中

**1 年後のゴール**
- 大田区および近隣で、個人事業主ヘルパーが「介護事業所の中抜きなしで」利用者と直接つながれる場
- プラットフォーム手数料は 10〜15%（既存事業所の 30〜50% 中抜きから大幅圧縮）
- 賃金 UP を仕組みに埋め込む（早朝/深夜のサージプライシング、地理近接案件の連続割当、レビュー連動の単価昇格）

**コア構成案**
- ヘルパー側: プロフィール / 空き時間カレンダー / 希望単価レンジ / 損害賠償保険証明 / eKYC
- 利用者・ご家族・ケアマネ側: 依頼ポスト（日時/所在地/必要資格/予算） / 候補ヘルパー選択 or 自動マッチング / 継続契約
- マッチングエンジン: PostGIS で地理近接 + 資格充足 + 単価レンジ重なり + 過去マッチ実績で並び替え
- 決済層: Stripe Connect or GMOペイメント、利用者→プラットフォーム→ヘルパーの直接送金
- 信頼: 両方向レビュー、賠償保険チェック、認証済み事業主バッジ
- 補助: 確定申告連携（既存「経費アプリ」のレシートデータを税務ソフトに流して節税自動化 → 実質手取り UP）

**法務・コンプラ留意点**
- 介護保険サービスは指定事業所が必要 → **自費（混合介護の自費部分）特化**で立ち上げるのが現実的
- 個人事業主と雇用の境界（偽装請負問題）に注意。指揮命令関係を作らない設計
- 損害賠償保険、個人情報保護、インボイス登録

**段階案**
1. 自費サービス × 既存ビレッジつばさ運営エリアで MVP（Phase 1+2: 登録 + 手動マッチング + 決済）
2. レビュー + 継続契約の信頼スコアを蓄積
3. 自動マッチングエンジン + サージプライシング
4. 福利厚生プール（健康診断・賠償保険・積立年金）

**判断ポイント（別チャットで深掘り中）**
- 法人形態（合同会社つばさ内 vs 別法人）
- 自費特化スタート vs 保険連携も視野
- 対象エリア第一弾、競合差別化軸（カイテク/アサインナビ等）
- 価格設定、ヘルパー獲得経路、利用者獲得経路、技術スタック、ブランド

**社内セルフマッチング（Phase 1 / 本リポ）との関係**
- 社内: ビレッジつばさ自社ヘルパーの空き時間を未割当案件に充てる。中抜き圧縮や単価交渉は対象外
- 外部: 個人事業主・複数事業者横断・賃金 UP・動的単価まで含むフルスコープ
- データモデル・技術スタックも別。流用するとしたら「ヘルパー × 利用者 × スケジュール」のドメインモデルくらい

**着手タイミング**
- 2026-05-06 時点では別チャットで構想練り中。社内セルフマッチング Phase 1 の運用が落ち着いてから本格設計（2026-Q3 以降想定）

**未解決課題**
- SaaS 化 vs IT 代行サービス化（最初は代行で実績、後で SaaS へ）
- 合同会社つばさ本業との並行リソース配分
- 社内セルフマッチングからの教訓をどう外部版に活かすか

### 2026-04-26 利用者情報マスタ + Google マップ連携機能

**きっかけ**
- 2026-04-26 の機能要望チャットで奥原さんから提案
- 現状: ヘルパーがスケジュール画面で利用者の住所を確認 → スマホで地図アプリに手入力 → ナビ起動 という手間が毎日発生
- 利用者の連絡先・特記事項も毎回紙やメモを参照している

**ゴール**
1. スケジュール画面で **利用者名タップ** → 利用者詳細画面（住所・連絡先・メモ・直近の支援履歴）が即表示
2. スケジュール画面で **行き先住所タップ** → Google マップ（or Apple マップ）でナビが即起動
3. ヘルパーが紙メモや別アプリを参照する手間を撲滅

**コア構成案**
- 新規テーブル `clients`
  - `id` (uuid PK), `beneficiary_number` (unique), `name`, `display_name`, `address`, `postal_code`, `phone`, `emergency_contact`, `notes`, `active`, `created_at`, `updated_at`
  - 既存 `client_users` / `clients` （`calm_check_targets.client_id` 参照先）と関係整理が必要
- 利用者マスタ管理画面 `/clients/`（一覧）/ `/clients/:id`（詳細・編集）
- 既存の `today-schedule` / `tomorrow-schedule` / `today-schedule-all` / `tomorrow-schedule-all` /
  `schedule-sync` の利用者名・行き先テキストにタップイベント追加
- Google マップ起動: `https://www.google.com/maps/dir/?api=1&destination=...`（iOS は自動でアプリ切替）

**実装フェーズ案**
1. **Phase A**: `clients` / `client_users` の既存テーブル調査（既存テーブルを使えるなら作り直し不要）
2. **Phase B**: 利用者マスタ画面 MVP（CRUD、認証は admin_users allow-list）
3. **Phase C**: 116名の利用者データ入力（CSV から一括 INSERT）
4. **Phase D**: 既存スケジュール画面に名前タップ → 詳細遷移を追加
5. **Phase E**: 行き先住所タップ → Google マップ起動を追加

**未解決課題・検討事項**
- 既存 `schedule.client` は文字列で利用者名が入っている → `clients` テーブルとの紐付けは「名前一致」「`beneficiary_number` 一致」のどちらにするか
- 116名分のデータ入力をどこから持ってくるか（受給者証データの活用？）
- 個人情報（住所・連絡先）の取り扱いは RLS 整備済みで OK だが、表示画面側で誰に何が見えるかの権限設計が必要

**着手タイミング**
- 機能1（スプレッドシート的スケジュール編集 HTML）の完成後に着手予定
- 2026-04-26 時点では機能1 を優先

### 2026-04-21 合同会社つばさ発「福祉業界向け IT オールインワン」構想

**きっかけ**

- 福祉業界は IT 特化製品が少なく、中小事業者は自作 or Excel で回していることが多い
- 合同会社つばさで運用している 3 アプリ構成（ヘルパー / 管理者 / 利用者）と自動化ルールは、他の福祉事業者にも転用できそう

**1年後のゴール**
- 大田区または近隣エリアの福祉事業者から「うちにも入れてほしい」と言われる状態に到達
- 営業活動はせず、業界コミュニティでの評判から引き合いが来る形を目指す

**3 段階の進め方**
1. **自社運用フェーズ（〜2026-10）**: 合同会社つばさで半年運用し、RULES.md / CHANGELOG.md / CURRENT_STATE.md / FUTURE_IDEAS.md / scripts/ の運用を洗練。村上翼さんとの 2 名体制でノウハウ蓄積

3. **外販フェーズ（2027-01〜）**: 最初は SaaS ではなく **IT ヘルパー代行サービス** として導入支援。奥原さん（＋ Claude）が導入を伴走する形。実績が数社たまったら SaaS 化を再検討

**コア構成案**

- 管理者向けダッシュボード（記録一覧・Excel 出力・研修管理・フィードバック対応）
- 利用者向けスケジュール提出アプリ

**現時点の未解決課題・検討事項**
- SaaS 化 vs 代行サービス化、どちらがスケールしやすいか
- 他社販売時の情報セキュリティ・個人情報保護（福祉は特に重要）の枠組み
- 合同会社つばさ本業（ヘルパー派遣）と並行して回せるリソース配分
